# i18n Design: LocaleManager App Localization

## Overview

Internationalize the LocaleManager app UI with en-CA and fr-CA support using react-i18next. Auto-detect browser locale with a sidebar toggle to override. Includes locale-aware date and number formatting.

## Library Choice

**react-i18next + i18next**

- `i18next` — core i18n framework
- `react-i18next` — React bindings (useTranslation hook, Trans component)
- `i18next-browser-languagedetector` — auto-detect browser locale

No server-side i18n needed — this is client-only.

## Translation File Structure

```
client/src/locales/
  en-CA/
    common.json       # shared: nav, buttons, labels, errors, statuses
    dashboard.json    # dashboard page strings
    project.json      # project dashboard, settings, new project, edit key
    editor.json       # translation editor page
    landing.json      # public landing page
  fr-CA/
    common.json
    dashboard.json
    project.json
    editor.json
    landing.json
```

### Key naming convention

Flat dot-notation keys within each namespace:

```json
{
  "nav.dashboard": "Dashboard",
  "nav.projects": "Projects",
  "actions.save": "Save",
  "actions.cancel": "Cancel",
  "errors.unauthorized": "You are logged out. Logging in again..."
}
```

### Namespace assignment

| Namespace   | Covers                                                        |
|-------------|---------------------------------------------------------------|
| common      | Nav labels, shared buttons (Save/Cancel/Delete), error/success toasts, status labels (Draft/In Review/Approved), form validation messages, role labels |
| dashboard   | Dashboard page: welcome message, stats cards, project list, empty states |
| project     | Project dashboard, new project, project settings, edit key, new key, import/export pages |
| editor      | Translation editor: search, filters, table headers, cell actions, status transitions |
| landing     | Landing page: hero, feature cards, sign-in                    |

## i18next Configuration

```typescript
// client/src/i18n.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';

// Import all translation files
import enCACommon from './locales/en-CA/common.json';
import enCADashboard from './locales/en-CA/dashboard.json';
import enCAProject from './locales/en-CA/project.json';
import enCAEditor from './locales/en-CA/editor.json';
import enCALanding from './locales/en-CA/landing.json';
import frCACommon from './locales/fr-CA/common.json';
import frCADashboard from './locales/fr-CA/dashboard.json';
import frCAProject from './locales/fr-CA/project.json';
import frCAEditor from './locales/fr-CA/editor.json';
import frCALanding from './locales/fr-CA/landing.json';

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: {
      'en-CA': {
        common: enCACommon,
        dashboard: enCADashboard,
        project: enCAProject,
        editor: enCAEditor,
        landing: enCALanding,
      },
      'fr-CA': {
        common: frCACommon,
        dashboard: frCADashboard,
        project: frCAProject,
        editor: frCAEditor,
        landing: frCALanding,
      },
    },
    fallbackLng: 'en-CA',
    defaultNS: 'common',
    detection: {
      order: ['localStorage', 'navigator'],
      lookupLocalStorage: 'i18nextLng',
      caches: ['localStorage'],
    },
    interpolation: {
      escapeValue: false, // React already escapes
    },
  });

export default i18n;
```

Imported in `client/src/main.tsx` before app renders:
```typescript
import './i18n';
```

## Language Switcher

A toggle in the sidebar footer, above "Sign Out":

```
[EN | FR]   ← simple segmented control
Sign Out
```

Implementation: `useTranslation()` hook's `i18n.changeLanguage('fr-CA')`. Persists to localStorage automatically via the language detector plugin.

Located in `client/src/components/app-sidebar.tsx`.

## HTML lang Attribute

Update `<html lang="...">` when language changes for accessibility. Use an `i18n.on('languageChanged')` listener or a `useEffect` in the app root to set `document.documentElement.lang`.

## Pluralization

Use i18next's built-in plural keys (`_one`, `_other` suffixes) for count-dependent strings like "Imported {{count}} translation(s)". Example:

```json
{
  "toast.importedCount_one": "Imported {{count}} translation",
  "toast.importedCount_other": "Imported {{count}} translations"
}
```

## Date and Number Formatting

Note: `date-fns` is already a project dependency but `Intl.DateTimeFormat` is preferred for locale-aware formatting since it's zero-bundle-cost and natively supports locale variants like `en-CA` vs `fr-CA`.

A `useLocaleFormat` hook in `client/src/hooks/useLocaleFormat.ts`:

```typescript
export function useLocaleFormat() {
  const { i18n } = useTranslation();
  const locale = i18n.language;

  return {
    formatDate: (date: Date | string) =>
      new Intl.DateTimeFormat(locale, { dateStyle: 'medium' }).format(new Date(date)),
    formatNumber: (n: number) =>
      new Intl.NumberFormat(locale).format(n),
    formatFileSize: (bytes: number) => {
      if (bytes < 1024 * 1024) {
        const kb = bytes / 1024;
        return `${new Intl.NumberFormat(locale, { maximumFractionDigits: 2 }).format(kb)} KB`;
      }
      const mb = bytes / 1024 / 1024;
      return `${new Intl.NumberFormat(locale, { maximumFractionDigits: 2 }).format(mb)} MB`;
    },
  };
}
```

Used in: document table (dates, file sizes), stats cards (numbers), anywhere dates appear.

## Validation Messages

Zod schemas use translation keys resolved at validation time, not at definition time. Approach: wrap schema creation in a function that takes `t`:

```typescript
// Before:
const schema = z.object({
  name: z.string().min(1, "Project name is required"),
});

// After:
function getProjectSchema(t: TFunction) {
  return z.object({
    name: z.string().min(1, t("validation.nameRequired")),
  });
}
```

This ensures messages update on language switch without remounting.

## Toast Messages

All toast calls in mutation callbacks use `t()`:

```typescript
// Before:
toast({ title: "Success", description: "Project created successfully" });

// After:
toast({ title: t("toast.success"), description: t("toast.projectCreated") });
```

## Status Labels

The status machine labels in `shared/schema.ts` stay as English constants (they're developer-facing). The UI maps them through i18n:

```typescript
// In components:
const statusLabel = t(`status.${translation.status}`); // "draft" → "Brouillon"
const transitionLabel = t(`transition.${from}-${to}`);  // "draft-in_review" → "Soumettre pour révision"
```

Translation files contain:
```json
{
  "status.draft": "Brouillon",
  "status.in_review": "En révision",
  "status.approved": "Approuvé",
  "transition.draft-in_review": "Soumettre pour révision",
  "transition.in_review-approved": "Approuver",
  "transition.in_review-draft": "À retravailler",
  "transition.approved-draft": "Rouvrir"
}
```

## What gets localized
- All page headings, descriptions, empty states
- Sidebar navigation labels
- Button labels (Create, Delete, Save, Cancel, etc.)
- Form labels, placeholders, validation messages
- Dialog/modal content (confirmations, instructions)
- Table headers
- Toast notifications (success/error)
- Status labels (Draft, In Review, Approved) and transition labels
- Date displays and file sizes
- Landing page marketing copy

## What stays English
- Translation key names (developer identifiers like `greeting.hello`)
- API error codes and server logs
- Console.log messages
- URL paths and query parameters

## Testing

Existing Playwright tests should continue to pass since en-CA is the fallback and all English strings remain the same. New tests can verify:
- Language toggle switches UI strings
- Persisted preference survives page reload
- French strings render correctly
